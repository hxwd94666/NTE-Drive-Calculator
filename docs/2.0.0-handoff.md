## 已完成

### 官方静态数据与可重复构建

- 构建器：`tools/game_data/build_static_database.py`。
- 当前官方来源根目录：`C:\work_files\documents\外部项目\HT\Content`。
- 发行库不保存完整来源 JSON；保留来源相对路径、文件哈希、行键和行内容哈希，支持更新审计。
- 每次游戏更新可用同一构建器重新生成 `data/game_static.sqlite3`，不依赖旧 `config/*.json`。

### 角色数据

- 官方角色目录：22 条。
- 角色六觉、三觉/六觉共鸣及明确的技能等级加成。
- 角色 1–80 级、六段突破前后基础生命/攻击/防御：1,806 条有效状态。
- 技能目录、技能类型、显示标记、升级等级条件与材料：84 个技能、756 条等级规则。
- `1056` 已明确为安魂曲 `1004` 的蝙蝠战斗形态：仅保留目录和规范角色关联，绝不生成独立成长、觉醒、技能或配装数据。

### 原始技能伤害参数

- `DT_SkillDamageData`：802 条官方伤害执行记录。
- `DT_SkillDamageGameplayModifyData`：14 条攻击倍率修正规则。
- 其中 599 条可按官方 `GAName` 关联至角色技能；其余属于其他系统技能，仍作为官方数据完整保留。
- DAO 的 `list_character_skills()` 已返回 `damage_entries`，但这些是原始数据，**尚未计算直接伤害**。

### 装备、弧盘与运行基础

- 静态库已包含官方装备属性、形状、套装、装备项、官方蓝图、弧盘目录和成长相关数据。
- `StaticGameDataDao` 是静态库唯一查询边界；`UserDataDao` 负责账号配置、稳定背包快照和方案。
- 已有 SQLite 背包投影和优化器入口；旧求解器仍存在兼容层，不能再把旧 JSON 当作新功能的权威数据源。

## 还缺什么

| 领域 | 缺口 | 结果影响 |
| --- | --- | --- |
| 直接伤害 | 尚未确认并实现伤害公式、各倍率数组的等级语义、暴击/防御/元素/破坏等结算顺序。 | 不能给出可信面板外伤害或以伤害为目标优化。 |
| Buff 与条件 | 普通觉醒、技能、套装、弧盘和战斗状态的 Buff 数值、触发条件、叠层与互斥规则尚未统一建模。 | 不能正确反映实战状态或六觉效果。 |
| 伤害归属 | 单个技能可能对应多段伤害记录；目前仅按 `GAName` 关联，尚无“哪一段何时命中”的技能动作定义。 | 不能计算连段、总伤害或循环。 |
| 用户养成 | 角色等级、突破、激活觉醒、技能等级与弧盘状态需要完整稳定地从账号快照进入新模型。 | 静态库虽完整，仍不能自动形成用户角色计算输入。 |
| 新 UI | 多数旧页面仍读取旧 JSON；角色养成、技能、Buff、伤害和新优化流程尚未完整迁入 SQLite DAO 架构。 | 用户无法在新数据链路中完成全流程操作。 |
| 配装闭环 | 全角色竞争分配、目标函数、结果解释和保存已具备部分基础，但尚未以可信伤害模型驱动。 | 当前推荐不能宣称是最终最优战斗配装。 |
| 安全执行 | 批量装配缺少可靠回滚与续跑；没有历史装备记录的新角色可能无法自动定位角色实例 UID。 | 一键执行仍需保守、可确认地推进。 |

## 阶段目标

### 阶段 1：数据基础（已完成）

验收条件：官方静态文件更新后，运行构建器即可生成 schema v7 SQLite；角色、觉醒、成长、
技能、伤害原始参数、装备和弧盘记录均有来源追溯，且发行库不含原始 payload。

### 阶段 2：可验证的角色战斗输入（下一阶段）

目标：将“用户当前角色”标准化为可计算输入，而不是散落在旧 JSON 与 UI 状态中。

小任务顺序：

1. 明确直接伤害公式及每个官方字段的语义、等级索引和结算顺序。
2. 建立独立的伤害模型与黄金样例测试；先覆盖一个角色的一项技能，不改 UI。
3. 建模角色等级、突破、技能等级、已激活觉醒和必要 Buff；与 v7 静态 ID 对齐。
4. 扩展到多段技能与角色专属条件，所有规则必须有可复现测试。

验收条件：给定固定角色养成、固定目标与固定战斗条件，计算结果可复现、可解释，并能定位到使用的官方静态记录。

### 阶段 3：以伤害为目标的配装

目标：在固定 `snapshot_id` 的背包快照上，为单角色和多角色生成可解释的配装方案。

关键要求：

- 形状、套装、官方蓝图和 UID 均使用官方 ID。
- 多角色竞争同一件装备时，先生成不可变候选集，再统一决策。
- 输出应说明属性变化、套装效果、伤害变化与未满足约束；不得只给一个黑箱分数。

验收条件：同一快照与同一配置重复运行结果一致，且不会在求解过程中修改背包。

### 阶段 4：新 UI、保存与安全执行

目标：将角色配置、伤害预览、方案比较和执行流程迁入新 SQLite/DAO 架构。

验收条件：用户可查看数据来源与计算条件，保存方案可回读；执行前有预检查、差异确认、失败记录以及可恢复机制。

## 终极目标

用户在本地同步一次稳定背包快照后，可以选择角色、等级/突破、技能、觉醒、弧盘和战斗条件；
项目基于官方静态数据和明确的公式，计算可解释的属性与伤害，并在不重复分配装备的前提下为一个或多个角色生成最优或可比较的配装方案。每个结果都应可追溯到固定快照、官方 ID、规则版本和数据集版本；实际装配操作始终经过预检查、用户确认和可恢复的执行记录。

## 协作约定

- 所有项目验证使用 Python 3.11：`py -3.11`。
- 静态库迁移只新增 `src/storage/sqlite/schema/NNN_*.sql`，不得修改已发布迁移的语义。
- 不将游戏官方原始文件、账号数据、日志、Token 或构建报告提交进仓库。
- 新数据功能只通过 DAO 访问 SQLite；旧 `config/*.json` 与 `my_roles_model.json` 仅用于旧页面兼容。
- 每个小任务只完成一个清晰边界：迁移/导入、计算规则、用户快照、优化器或 UI 之一；附带最贴近的单元测试。
- 源码改动至少执行：

```powershell
py -3.11 -m unittest tests/test_encoding_guard.py
git diff --check
```

- 静态数据改动额外执行：

```powershell
py -3.11 -m unittest tests.test_static_game_database tests.test_static_game_data_dao -v
```
