# 2.0 扩展指南

## 增加页面

在 `src/features/<feature>/page.py` 实现页面工厂，再向 `src/ui/navigation.py` 的导航定义增加一项。页面不要自己创建数据库或 nte-core 进程；由应用外壳传入 DAO、服务或只读状态。

主页快捷入口也应使用导航 key，而不是写死堆叠页面的数字下标。这样调整页面顺序时，按钮仍会打开正确页面。

## 增加静态数据

开发者更新游戏官方文件后，先在项目外整理来源清单，再运行 `tools/game_data` 下的导入工具生成候选数据库。检查步骤至少包括：

1. schema 版本和来源文件摘要；
2. 角色、装备、套装、形状和属性之间的外键关系；
3. 小游戏、测试配置和重复技能配置的明确分类；
4. `tools/game_data/inspect_static_database.py` 的摘要与抽查；
5. 替换 `data/game_static.sqlite3` 后运行完整测试。

运行程序不负责自动获取或处理游戏文件。仓库只维护程序实际需要、来源清楚且体积受控的发行数据。

## 增加 DAO 查询

只读官方数据加到 `StaticGameDataDao`；账号状态加到 `UserDataDao`。服务层不能直接拼接 SQL。计算任务应在开始时记录一个 `snapshot_id`，然后始终显式传入该 ID。

用户库变更要新增迁移 SQL，不能修改已发布迁移的含义。迁移后同时更新 schema 版本、旧库迁移测试、新库初始化测试和打包资源测试。

快照清理必须通过 `UserDataDao.prune_inventory_snapshots`，不得直接删除表记录。
该方法会保护当前快照、最近保留窗口、所有装配方案引用的来源快照和未完成装配任务
的来源快照；需要新策略时
应先扩展保护条件与测试，避免使已保存方案失去可复现输入。

## 扩展新配装算法

现有兼容算法协议在 `src/optimizer/contracts.py`。2.0 新算法优先复用 `SqliteLoadoutOptimizer` 已建立的官方 ID 输入与输出格式：

- 输入：角色 ID、固定快照 ID、官方属性 ID 权重；
- 约束：官方蓝图、形状、套装需求和不可重复 UID；
- 输出：原生 UID、目标行列、总分、来源快照和算法版本。

要加入角色边际或战斗收益时，把边际计算实现为独立评分器，通过官方 `property_id` 返回权重或收益，不要把中文显示名重新写进算法。旧显示名称兼容仍只留在 `src/domain/stat_catalog.py`。

全角色优化还需要处理同一装备被多个角色竞争。建议先构造不可变候选集，再由分配器统一决策，最后一次性保存每个角色的方案；不要让角色循环直接修改当前背包状态。

## 扩展背包同步方式

设置页保存的是同步方式 key。新实现应满足与 `InventorySyncService` 相同的状态语义：等待、收集中、保存中、稳定监听和错误。无论数据来自哪种同步方式，进入用户数据库前都必须是完整快照，并经过内容稳定判断。

不要使用“历史最大条数”或某个玩家的装备数量作为完成条件。稳定条件只取决于当前完整事件的内容和安静时间。

## 扩展装配方式

装配器消费保存后的方案，不应重新运行优化。新的装配方式至少要实现：

- 调用前确认方案装备仍存在于当前稳定快照；
- 明确角色实例；
- 防止重复 UID 和越界位置；
- 调用后取得可验证的新状态；
- 失败时返回可展示的错误，不静默假定成功。

`EquipmentApplyService` 已实现 nte-core 一键装配的这一闭环。手柄方式可以保留为另一实现，但应输出相同的验证结果结构。

批量装配必须使用 `equipment_apply_job`、`equipment_apply_job_item` 和
`equipment_apply_job_log` 持久化进度。失败时只可重试失败项及其后的 pending 项，不能把
“已发送指令”误记为成功；回滚必须作为独立、可验证的后续能力实现。

## 测试与提交

每个功能提交前至少运行对应单元测试、`tests/test_encoding_guard.py` 和 `git diff --check`。准备发布时运行：

```powershell
python -m unittest discover -s tests
python -m compileall -q src tests
python build_installer.py --generate-only
```

建议按“同步与存储”“主页与设置”“资源”“优化器”“装配”“文档与发布”拆分提交。这样出现问题时可以定位到单一边界，也方便在 `2.0.0` 分支逐步验收。
