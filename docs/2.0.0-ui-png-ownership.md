# 2.0 页面 PNG 资源归属

## 结论

不要把所有 PNG 物理放入同一个目录。应当统一**引用入口和清单**，但按资源的生命周期保留不同目录：

- `assets/game_ui/`：随程序发布、只读、用于 2.0 页面展示的轻量 UI 图片。以 `manifest.json` 和 `GameUiAssetCatalog` 为唯一引用入口；键使用官方 `character_id`、`item_id` 或属性 ID。
- `config/templates/`：识别、自动操作和旧页面兼容模板。运行时配置目录可写，且已有 `BUNDLED_CONFIG_DIR` 兜底，不能与 `assets/` 合并。
- `accounts/<account_id>/scanned_images/` 与 `accounts/<account_id>/`：用户截图、剪贴板临时图和扫描产物；绝不能作为随包 UI 资源管理。
- `config/github/`：文档/发布配图；后续若仍在使用，应迁到相应文档目录，而不是并入应用 UI 资源。

当前 `assets/game_ui/manifest.json` 已经是新页面资源的正确中心；它不应被旧模板或用户数据污染。

## 目录职责

```text
assets/
  app_icon.*
  game_ui/
    manifest.json                 # 2.0 页面唯一的资源映射清单
    characters/                   # official character_id -> PNG
    attributes/                   # official attribute_id -> PNG
    equipment/
      core/                       # official item_id -> PNG
      module/                     # official module item_id -> PNG
    forks/                        # official fork_id -> PNG
    monsters/                     # static_table + monster_id -> PNG

config/
  templates/
    recognition/                  # 未来可收纳 new_tag 等识别专用模板
    shapes/                       # 未来可收纳旧版驱动形状模板
    roles/                        # 旧角色页与自动操作共用的兼容模板
    guide/                        # 可更新的新手引导图片

accounts/<account_id>/
  scanned_images/                 # 用户扫描截图，禁止打包和迁移至 assets
```

上面的 `recognition/` 与 `shapes/` 是目标结构，不应在旧调用仍按平铺文件名读取时直接移动。先提供兼容解析，再逐步迁移调用。

## 引用规则

1. 新建 2.0 页面不得直接拼接 `assets/.../*.png` 路径；必须经 `GameUiAssetCatalog` 按官方 ID 获取。
2. 新页面不得读取 `my_roles_model.json` 或旧角色名来定位头像；使用官方 `character_id`。仓库页应使用现有 `equipped_character_id`，而不是名称回退。
3. 识别、自动操作所需模板只能从 `runtime.TEMPLATE_DIR`（并保留 bundled fallback）读取；它们不是 UI 静态资产。
4. 账号截图和临时图片只能从账号目录读写，不加入 `assets/`、manifest 或安装包静态资源。
5. 若 2.0 页面确实需要新的随包 PNG，先由资源构建工具生成压缩图，再写入 `assets/game_ui/manifest.json` 和对应测试；不要直接复制原始大图。

## 建议迁移顺序

1. 保持现有文件位置，先把仓库页的角色头像和核心图标全部切换到 manifest。
2. 为 2.0 所需的驱动形状补充 `game_ui/equipment/module`（或保留 Qt 绘制占位），使新页面不再回退 `config/templates`。
3. 把旧页面的模板读取收敛到一个兼容解析器，再把 `config/templates` 内部拆为 `recognition`、`shapes`、`roles`、`guide`。
4. 当 `execute`、`equipment`、`identify`、`blueprint` 和旧 `features/role` 退役后，再删除旧模板兼容分支；不要在 2.0 迁移期间复制两套权威图源。
